# feedback_writing.md — 文体FB蓄積

## 2026-04-03：フリーランスプログラマー記事FBより抽象化

### 1. 統計数値は「出典名（取得時点）」を文末に必ず付記
年収・単価・割合・料金などの数値を記載する場合、同一文内または直後に出典名と取得時点を明記する。

**NG:** 「フリーランスプログラマーの年収相場は500万〜700万円です。」
**OK:** 「フリーランススタート（エン・ジャパン、2025年9月時点）によると、フリーランスエンジニアの平均単価は月76.0万円です。」

### 2. 複数ソースからの推計レンジは計算方法を示す
編集部が複数データから算出した数値レンジ（「500万〜700万円」等）は、根拠となるデータと計算方法を本文に明示する。明示できなければ、各データを個別に出典付きで列挙する。

### 3. 数値の出典は一次情報を優先
参照する情報の優先順位:
1. 公式サイト（サービス提供元）
2. 官公庁（厚労省・国税庁・消費者庁等）
3. 原典PDF（調査報告書・白書）
4. プレスリリース
5. メディア記事（まとめサイトは補助のみ）

まとめサイト・比較サイトの数値をそのまま転載しない。必ず一次情報まで遡って確認する。

### 4. 別職種・別条件のデータを流用しない
「Webデザイナー向けの単価データ」を「プログラマー全体の相場」に使う等、対象が異なるデータの流用は禁止。職種・条件・集計方法が記事のテーマと一致するか必ず確認する。

### 5. 確認できなかった数値は書かない
裏取りで確認できなかった数値（サイト上で見つからない、数値が変わっている等）は本文に記載しない。「公式サイトで要確認」と書くか、確認できた別の数値に差し替える。

## 2026-04-05：文章構造FBより抽象化

### 6. 1文目と2文目が裏表の繰り返しにならないようにする
「〜が必要です」→「〜しないと失敗します」のように、同じ内容を肯定と否定で繰り返さない。
2文目は1文目の**根拠**か**具体例**にする。

**NG:** 「実務経験が必要です。経験がないと案件を受注できません。」
**OK:** 「実務経験が必要です。エージェント案件の多くは『実務経験2〜3年以上』を受注条件に設定しています。」

### 7. 同じ角度の文が3連続しない
同じ動詞・同じ構文（「〜してくれる」「〜できる」等）が3文以上続くと読者が飽きる。
3連続になりそうなら、ul/liに統合して1回で済ませる。

### 8. strong締めは本文にない情報を入れる
strong締めが直前の本文と同内容になるパターンが頻出。
strong締めには**本文に書いていない具体的行動**か**別角度の価値**を入れる。

**NG:** 本文「評価がたまると受注率が上がる」→ strong「評価が貯まったらエージェントに移行」（ほぼ同じ）
**OK:** 本文「評価がたまると受注率が上がる」→ strong「最初の5件は赤字覚悟で評価を集め、6件目から単価を上げてみてください」（新しい行動）

### 9. H2導入文は「宣言だけ」で終わらせない
「〜が前提です」「〜が重要です」だけで終わる導入文は弱い。
リスク・必要性・読者メリットを具体的に1文添えてから本題に入る。

**NG:** 「フリーランスへの独立は、スキル習得が前提です。」（宣言だけ）
**OK:** 「スキルが不十分なまま独立すると、案件の単価が上がらず生活費を下回るリスクがあります。」（リスク提示）

### 10. 文章間のつながりを論理軸・時間軸で整理する
1つのH3内で話題が毎文飛ぶ場合、時間軸（何がなくなるか→毎月の負担→将来の備え）や論理軸（原因→結果→対策）で文を並べ替える。

### 11. 箇条書きの羅列を一文に詰め込まない
3項目以上の列挙は一文に詰め込まず、ul/liに分解する。
全H3に最低1つの表またはul/liを入れると読みやすくなる。

**NG:** 「ポートフォリオの作成・更新、クライアントとの面談、契約書の確認、請求書の発行、入金管理、確定申告の準備などが発生します。」
**OK:**
```
フリーランスのプログラマーが対応する業務は以下の通りです。
<ul>
<li>ポートフォリオの作成・更新</li>
<li>クライアントとの面談</li>
<li>契約書の確認</li>
<li>請求書の発行・入金管理</li>
<li>確定申告の準備</li>
</ul>
```
