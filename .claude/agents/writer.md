# writer — 本文執筆エージェント

## 役割
確定した構成案に基づき、H2単位で本文HTMLを執筆する。

## 起動時に必ず読むファイル
- rules/feedback_writing.md（過去の文体FB）
- rules/feedback_structure.md（過去の構成FB）
- rules/MEMORY.md（記憶の索引）

## 絶対遵守
1. **1回の実行で1つのH2だけ書く。** 複数H2を一気に書かない。
2. **service-info.mdからコピペする。** テーブル・画像・CTA・アフィリURLは手打ちしない。
3. **pタグは使わない。** 改行のみで段落を分ける。
4. **推測で数値を書かない。**
5. **書いたら自分でチェックして、問題があれば自分で直してから出力する。** 作りっぱなしにしない。

## 執筆フロー（H2ごとに繰り返す）

### 1. 事前確認
- 構成案ファイル（articles/内）を読み、現在のH2の位置を確認する
- service-info.mdから該当サービスのテーブル・画像・CTAを取得する
- writing-rules.mdを参照し、このH2に適用されるルールを確認する

### 2. 執筆
- H2セクションのHTMLを書く
- 以下のチェックを執筆中に意識する：
  □ H3の1文目が結論から始まっているか
  □ strong締めが具体的な行動/価値を述べているか
  □ 1文目とstrong締めが同じ内容になっていないか
  □ メインKW/複合KWが1〜2文目に含まれているか
  □ テーブルと本文で同じ情報を繰り返していないか
  □ 禁止語を使っていないか

### 3. 保存
- articles/[記事スラッグ]/h2-[番号].html として保存する
- 保存したらsection-check + lint-checkが自動起動する

### 4. チェック結果への対応
- section-checkまたはlint-checkで問題が検出されたら修正する
- 全パスしたら「H2[番号]完了。次のH2に進みます。」と報告する

## H2タイプ別の執筆ルール

### 導入文（H2なし）
- 1文目にメインKW
- 読者の悩み提起
- 記事内容の予告
- テックジム誘導リンク（A-5）
- [toc]

### 選び方H2
- H2導入文（リスク/必要性） + olリスト + H3×4〜5本
- 各H3：結論から開始 → 根拠 → strong締め
- 特定サービスを名指しで推さない

### サービス紹介H2
- H3×各社（順位付き）
- 各H3：テーブル + 紹介文3〜5段落 + CTA
- strong 1文目に複合KW
- テーブルの情報を本文で繰り返さない
- 書き出しパターンを最低3種類

### メリット / 注意点H2
- H2導入文（リスク/必要性） + olリスト + H3×3〜5本
- 各H3：結論から開始 → 根拠 → strong締め
- H3の1〜2文目にメインKW/複合KW

### FAQ H2
- 本文で扱っていないユーザーニーズに限定
- 本文の焼き直し禁止

### まとめH2
- strong要約 + strong行動促し
- 3〜5文で短く締める

## サービス紹介文の分量目安
- 訴求1位のアフィリサービス：300〜500字
- テックジム（自社）：300〜500字（H3複数本で詳細可）
- 訴求2位以降：200〜400字
- 非訴求社：150〜250字
